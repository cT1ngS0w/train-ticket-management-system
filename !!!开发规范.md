# Unified Project Spec

This document is the team-wide source of truth for structure, database rules, and module boundaries.

If any earlier draft in `README.md` or chat messages conflicts with this file, this file wins.

## 1. Project Stack

- Language: C++17
- UI framework: Qt Widgets
- Build system: CMake
- Database engine: SQLite
- Database access library: Qt SQL (`QSqlDatabase`, `QSqlQuery`)
- Database file name: `train_ticket.db`

## 2. Required Architecture

The project must follow this dependency direction:

`UI -> Module Manager -> DatabaseManager -> SQLite`

Rules:

- UI code must not execute SQL directly.
- `DatabaseManager` is the only class allowed to open connections and execute SQL.
- `LoginManager`, `TrainManager`, `TicketManager`, and `StatisticsManager` contain module logic.
- Module code may call `DatabaseManager`, but must not bypass it.

## 3. Canonical File and Class Structure

The repo structure must stay consistent with the current project layout:

- `include/database_manager.h`
- `include/login_manager.h`
- `include/train_manager.h`
- `include/ticket_manager.h`
- `include/statistics_manager.h`
- `include/main_window.h`
- `src/database/`
- `src/login/`
- `src/train/`
- `src/ticket/`
- `src/statistics/`
- `src/ui/`

Mandatory naming rule:

- Keep the existing `*Manager` naming.
- Do not rename existing folders or core classes without team discussion.
- Do not create a parallel architecture with different top-level names such as `AuthService` or `TrainService` unless the manager explicitly approves a refactor.

## 4. Ownership Boundaries

- Each developer should stay inside their assigned module unless a shared interface change is required.
- Shared interface changes must be discussed before merging.

## 5. Database Schema V1

This project uses a simplified schema for the course scope.

Important decision:

- In V1, there is no separate `Ticket` table.
- One row in `Order` represents one purchased ticket.
- This is intentional and simplifies booking, refund, change, history, and statistics.
- If seat assignment, multi-passenger orders, or unique printable ticket records are required later, a `Ticket` table can be added in V2.

### 5.1 User

| Column | Type | Constraints | Meaning |
| --- | --- | --- | --- |
| userId | INTEGER | PRIMARY KEY AUTOINCREMENT | User ID |
| username | TEXT | UNIQUE NOT NULL | Login name |
| password | TEXT | NOT NULL | Password |
| role | INTEGER | NOT NULL | `0=Guest`, `1=Seller`, `2=Admin` |
| enabled | INTEGER | NOT NULL DEFAULT 1 | Account status |

### 5.2 Station

| Column | Type | Constraints | Meaning |
| --- | --- | --- | --- |
| stationId | INTEGER | PRIMARY KEY AUTOINCREMENT | Station ID |
| stationName | TEXT | UNIQUE NOT NULL | Station name |

### 5.3 Train

| Column | Type | Constraints | Meaning |
| --- | --- | --- | --- |
| trainId | INTEGER | PRIMARY KEY AUTOINCREMENT | Train ID |
| trainNumber | TEXT | UNIQUE NOT NULL | Train number |
| departureStationId | INTEGER | NOT NULL | FK to `Station.stationId` |
| arrivalStationId | INTEGER | NOT NULL | FK to `Station.stationId` |
| departureTime | TEXT | NOT NULL | Departure time text |
| arrivalTime | TEXT | NOT NULL | Arrival time text |
| totalSeats | INTEGER | NOT NULL | Total seats |
| remainingSeats | INTEGER | NOT NULL | Remaining seats |
| enabled | INTEGER | NOT NULL DEFAULT 1 | Whether train is active |

Required constraints:

- `FOREIGN KEY(departureStationId) REFERENCES Station(stationId)`
- `FOREIGN KEY(arrivalStationId) REFERENCES Station(stationId)`
- `remainingSeats >= 0`
- `totalSeats >= 0`
- `remainingSeats <= totalSeats`

### 5.4 Order

| Column | Type | Constraints | Meaning |
| --- | --- | --- | --- |
| orderId | INTEGER | PRIMARY KEY AUTOINCREMENT | Order ID |
| userId | INTEGER | NOT NULL | FK to `User.userId` |
| trainId | INTEGER | NOT NULL | FK to `Train.trainId` |
| passengerName | TEXT | NOT NULL | Passenger name |
| purchaseTime | TEXT | NOT NULL | Purchase timestamp |
| status | INTEGER | NOT NULL | `0=Booked`, `1=Refunded`, `2=Changed` |

Required constraints:

- `FOREIGN KEY(userId) REFERENCES User(userId)`
- `FOREIGN KEY(trainId) REFERENCES Train(trainId)`

## 6. Module-to-Table Access Rules

Allowed read/write ownership:

- `DatabaseManager`: all tables, all SQL
- `LoginManager`: business logic for `User`
- `TrainManager`: business logic for `Train` and `Station`
- `TicketManager`: business logic for `Order` and train seat changes
- `StatisticsManager`: read-only statistics from `Order` and `Train`
- UI classes: no direct table access, no SQL

Non-negotiable rule:

- Only `DatabaseManager` writes SQL statements.

## 7. Required Responsibilities by Module

### 7.1 DatabaseManager

`DatabaseManager` must own:

- database connection setup
- database initialization
- table creation
- shared query execution helpers
- transaction handling when needed

Recommended responsibilities:

- open/connect database
- close database
- initialize schema
- execute prepared queries

### 7.2 LoginManager

`LoginManager` should handle:

- username/password validation
- role checking
- admin login
- seller login
- guest access logic
- create seller account
- modify password
- enable or disable account

### 7.3 TrainManager

`TrainManager` should handle:

- add train
- update train
- delete or disable train
- station management
- search by train number
- search by station
- seat inventory configuration

### 7.4 TicketManager

`TicketManager` should handle:

- book ticket
- refund ticket
- change ticket
- remaining seat inquiry
- order history lookup

When booking or refunding:

- booking decreases `Train.remainingSeats`
- refund increases `Train.remainingSeats`

### 7.5 StatisticsManager

`StatisticsManager` should handle read-only reporting such as:

- sales statistics
- monthly passenger flow
- popular route ranking
- order-based summaries

## 8. AI Assistant Rules For Every Teammate

If any teammate uses AI to generate code, that AI must follow these rules:

- Follow this file as the global project contract.
- Keep the existing repo structure and class names.
- Do not invent a different architecture.
- Do not add direct SQL into UI code.
- Do not create new top-level modules unless explicitly requested.
- Do not rename database fields defined in this spec.
- Do not modify another teammate's owned module unless the task explicitly requires a shared-interface change.
- If a shared interface must change, change the minimum possible surface and clearly explain it in the PR.

## 9. Integration Rules

Before opening a PR:

- pull latest `develop`
- make sure the project still builds
- keep changes limited to the assigned scope
- write clear commit messages
- explain any dependency on another module

Every progress update should include:

- `Done`
- `Next`
- `Blocked`

## 10. Scope Control

For the first integrated version, prioritize:

- login and roles
- train CRUD
- station CRUD
- search
- booking
- refund
- change
- order history
- statistics based on orders and trains

Defer for the moment:

- seat numbering
- multi-passenger booking per order
- payment workflow
- operation logs
- advanced routing algorithms

## 11. Final Priority

Consistency is more important than sophistication.

All teammates should prefer:

- the same table names
- the same field names
- the same class names
- the same directory layout
- the same access rules

This is the standard all contributors and all teammate AIs must follow.
